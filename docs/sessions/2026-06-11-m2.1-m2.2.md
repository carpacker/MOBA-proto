## Cross-cutting decisions (the calls that shaped the code)

These are the decisions that weren't obvious from the roadmap and required a judgment call.

### ADR-0012 — commit to Vulkan 1.3 dynamic rendering + sync2, no fallback
M2.0 deliberately left this gate open. M2.1's first pipeline forced it. The choice:
make `dynamicRendering` + `synchronization2` a **hard minimum spec**, gated at device
selection, with **no render-pass / sync1 fallback path**. Rationale: synchronization is
the #1 defect source in this phase, and a fallback means writing every barrier *twice*.
One barrier vocabulary is worth the lost hardware reach (pre-2022 drivers) for an engine
that ships years out. Recorded as ADR-0012 so the decision is cited, not re-litigated.

### The arena hard-abort policy is for *budgets*, not *untrusted input*
This was the throughline of **both** reviews and the single most important engineering
lesson of the session.

- **M2.1 (pipeline cache):** the review found that reading an oversized on-disk cache
  file straight into the renderer's fixed arena would hit the always-on `ENSURE` overrun
  abort — a corrupt file becomes a startup crash-loop. Fixed with `platform_file_size` +
  an 8 MiB bound *before* the read.
- **M2.2 (texture):** I applied what *looked like* the same fix — bound the TGA file at
  32 MiB before reading. The M2.2 review caught that it was **wrong**: the raw file and
  its decoded RGBA8 share the *same* arena, and 24bpp (3 B/px) decodes to 4 B/px, so
  worst-case live bytes ≈ 7/3 × file. A valid file between ~27 and 32 MiB passed the gate
  and *then* aborted. Three independent lenses flagged it. Fixed by **deriving the cap
  from the arena size** (3/8, below the 3/7 worst case) so the invariant holds by
  construction.

The principle, now load-bearing: **a fixed-arena size bound must cover every allocation
that lands in the arena, not just the first one.** ADR-0009's two-channel rule
(programmer-bug → abort; expected failure → degrade gracefully) means any on-disk size
must be incapable of reaching the abort path. Getting this subtly wrong *twice* — once
caught by review, once where my own "fix" reproduced the class — is exactly why the
adversarial pass earns its cost.

### `set=0` left deliberately empty
The quad's descriptor layout is `[set=0 empty placeholder, set=1 material]`, even though
M2.2 only uses set 1. Reserving set 0 now means M2.3's per-frame view/proj UBO drops in
**without** recreating the pipeline layout or renumbering the material set. A small piece
of forward-design that the roadmap's rung order justifies.

### Provisional seams, marked as such
`renderer_upload_texture` is a single-texture, blocking, startup-only seam — explicitly
documented as **provisional until M2.5** unifies upload into typed handles with
deferred-destroy. Same for the app-layer TGA decoder (→ M4.0 asset module) and the
absolute `MOBA_ASSET_DIR` / `MOBA_SHADER_DIR` paths (→ Phase 4 user-dir story). The
pattern: ship the simplest thing that satisfies the rung, mark the seam, cite the
milestone that folds it in. A review finding corrected one such marker (M4.0 is the
*runtime* parser; the cooker is M4.1) — milestone pointers are part of the contract trail.

### vk_alloc Phase 1: naive on purpose, with a tripwire
One dedicated `VkDeviceMemory` per resource. Deliberately the dumbest allocator that
works — and *because* buffers/images never share memory, `bufferImageGranularity` is a
non-issue, and because staging is `HOST_COHERENT`, there's no flush math anywhere. The
live allocation count is hard-`ENSURE`d under 3500 (below the 4096
`maxMemoryAllocationCount` floor): when that fires, it's the signal the naive scheme's
time is up and the block allocator (a later, deliberate phase) is due. The cap *is* the
design note.

## The adversarial review process — what worked, and a real lesson

Each milestone got a multi-lens workflow: 5 lenses (Vulkan sync, allocator/IO, build,
seam/architecture, tests), each producing findings, each finding then attacked by **2
independent skeptics** told to *refute* it and default to "refuted" when unsure. Only
findings both skeptics confirm get fixed.

**It paid for itself, repeatedly.** Across the two milestones it caught: the multi-config
shader-output clobber (proven from `.ninja_log`), both arena-abort bugs above, an
untested atomic-write failure branch, uninitialized-read UB in two tests, an untested
UTF-8 path contract, a `maxImageDimension2D` validation gap (a `vkCreateImage` UB
reachable from a well-formed TGA), and a stride-regression test gap a skeptic *proved* by
mutation (the bad `width*4` stride passed in Release; only Debug's `/RTC1` stack-fill
caught it incidentally). Most of these are the kind of latent, plausible-looking bug that
ships green and bites a month later.

**The lesson — a truncated review is not a clean review.** The M2.2 review's first run hit
the API session limit partway through: the three Vulkan/seam lenses never ran, and their
absent findings plus the failed skeptics defaulted the tally to a misleading *0 confirmed
/ 8 refuted*. Taking that at face value would have merged the major arena bug. **Resuming**
the same workflow (cached lenses return instantly; only the dead agents re-run) produced
the real result — 5 confirmed, including the major one. The takeaway now in the journal:
a review cut short by the session limit reports "refuted" as an *artifact of not having
run*, not a verdict. Always resume before trusting a review that logged limit failures.

## Recurring engineering themes reinforced this session

- **First real use of scaffolding finds the bugs.** `add_shader_library` was written back
  in M0.2 and never invoked until M2.1 — and held two latent defects (an empty genex
  argument glslc read as a second input file; config-shared `.spv` outputs that
  ping-ponged under Ninja Multi-Config). Code that has never run is unproven code.
- **Read back before present.** A presented swapchain image belongs to the presentation
  engine; the readback copy goes *between* `CmdEndRendering` and present, inside the
  fence-covered submission. `HOST_COHERENT` means the fence wait alone makes it visible.
- **One-shot upload visibility is barrier-side, not fence-side.** `end_one_shot`'s fence
  only tells the *host* the copy finished; later frame submissions see the data because
  the `barrier2` into `VERTEX_ATTRIBUTE_INPUT` / `SHADER_READ_ONLY` was recorded *inside*
  the one-shot.
- **CRT narrow APIs are ANSI, not UTF-8.** `std::remove()` can't delete a file created via
  a UTF-8 path; tests clean up with `_wremove`. Caught by a skeptic *inside its own
  suggested fix*.
- **`/RTC1` can mask a missing test.** A test that passes only because Debug fills the
  stack with `0xCC` is not real coverage — pin the behavior explicitly so Release agrees.
- **The screenshot loop is what makes the renderer verifiable at all.** The agent can't
  capture the live window (isolated window station), so `sandbox --screenshot` + BMP→PNG
  is the only session-independent visual proof. Both milestones' DoDs rest on pixel-exact
  reads of that capture (the triangle's RGB corners; the quad's R/G/B/Y UV markers and
  bit-exact sRGB 255/240/32 round-trip).

## Process / workflow notes

- **Every change through a PR, even docs.** Feature PR per milestone (squash), then a
  *separate* docs PR (journal + README) branched off `main` so it lands independently and
  never conflicts with the feature branch. The pre-push hook runs the full `/WX` build +
  ctest gate on every push, so red never reaches the remote.
- **The gate ran locally before every push and CI mirrored it** — green on Debug +
  Release for all four PRs. CI builds the *null* backend (no SDK on the runner); the
  Vulkan path is verified locally, which is why the seam (`renderer.h` Vulkan-free, the
  pipeline-cache checker and TGA decoder compiled headlessly) matters.
- **ASCII-only batch/commit discipline held.** Commit messages and PR bodies via
  `-F`/`--body-file`, never PowerShell here-strings.
